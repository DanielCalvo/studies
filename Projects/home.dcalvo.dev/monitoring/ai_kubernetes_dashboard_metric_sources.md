# Kubernetes Dashboard Metric Sources

The four Kubernetes Views dashboards use one Grafana datasource: Prometheus.
Prometheus builds that datasource by scraping three metric producers in the
cluster.

## Summary

| Metric producer | How it runs | What it exports | Mainly used for |
| --- | --- | --- | --- |
| kube-state-metrics | Deployment in `monitoring` | Kubernetes object state and configuration | Namespaces, pods, deployments, desired/current replicas, status, restarts, requests, and limits |
| node-exporter | DaemonSet in `monitoring`, one pod per node | Linux operating-system and hardware metrics | Node CPU, RAM, load, filesystems, disks, and network interfaces |
| kubelet/cAdvisor | Built into the kubelet on every k3s node | Actual pod and container resource usage | Pod/container CPU, memory, network, filesystem, throttling, OOM, and PVC usage |

These are metric producers, not three Grafana datasources. Grafana only queries
the `Prometheus` datasource. Prometheus scrapes and stores the metrics from all
three producers.

```text
kube-state-metrics ----\
node-exporter ----------> Prometheus ----> Grafana Kubernetes dashboards
kubelet/cAdvisor -------/
```

## 1. kube-state-metrics

kube-state-metrics watches the Kubernetes API. It describes what Kubernetes
objects exist, what they request, and their current control-plane state. It does
not measure actual container CPU or memory consumption.

Examples:

```promql
kube_node_info
kube_pod_info
kube_pod_status_phase
kube_pod_container_status_restarts_total
kube_pod_container_resource_requests
kube_pod_container_resource_limits
kube_deployment_status_replicas_available
kube_deployment_status_replicas_unavailable
```

Cluster configuration:

- Workload: `monitoring/kube-state-metrics` Deployment.
- Service: `monitoring/kube-state-metrics`, port `http`.
- Scrape definition: `prometheus_operator/ksm-servicemonitor.yaml`.
- Helm release: `kube-state-metrics`.

Its ServiceMonitor uses `honorLabels: true` to preserve the real Kubernetes
`namespace` and `pod` labels and attaches `cluster="homelab"` for the dashboard
cluster selector.

## 2. node-exporter

node-exporter reads Linux kernel and operating-system statistics from each
Orange Pi. It reports node health, not Kubernetes object state or per-container
usage.

Examples:

```promql
node_cpu_seconds_total
node_memory_MemAvailable_bytes
node_load1
node_filesystem_avail_bytes
node_disk_read_bytes_total
node_disk_written_bytes_total
node_network_receive_bytes_total
node_network_transmit_bytes_total
```

Cluster configuration:

- Workload: `monitoring/prometheus-node-exporter` DaemonSet.
- Shape: one exporter pod on `opi1` and one on `opi2`.
- Service: `monitoring/prometheus-node-exporter`, port `metrics` on 9100.
- Helm values: `node-exporter/values.yaml`.
- ServiceMonitor: generated by the `prometheus-node-exporter` Helm chart.

The Helm values add `cluster="homelab"` to its samples for the dashboard cluster
selector.

## 3. kubelet and cAdvisor

cAdvisor is embedded in each kubelet; there is no separate cAdvisor Deployment
or DaemonSet to install. Prometheus scrapes the kubelets on `opi1` and `opi2`
through the Prometheus Operator-managed `kube-system/kubelet` Service.

The two scrape paths serve different groups of metrics:

| Path | Purpose |
| --- | --- |
| `/metrics` | Kubelet health, running pod/container counts, and PVC volume usage |
| `/metrics/cadvisor` | Actual container CPU, memory, filesystem, and network usage |

Examples:

```promql
kubelet_running_pods
kubelet_volume_stats_used_bytes
kubelet_volume_stats_capacity_bytes
container_cpu_usage_seconds_total
container_memory_working_set_bytes
container_network_receive_bytes_total
container_network_transmit_bytes_total
container_cpu_cfs_throttled_seconds_total
container_oom_events_total
machine_cpu_cores
machine_memory_bytes
```

Cluster configuration:

- Runtime: built into the kubelet on every k3s node.
- Service: `kube-system/kubelet`, port `https-metrics` on 10250.
- Scrape definition: `prometheus_operator/kubelet-servicemonitor.yaml`.
- Authentication: Prometheus service-account bearer token.
- TLS: certificate verification is disabled for this trusted home network.

Both kubelet endpoints use `honorLabels: true` so the original namespace, pod,
and PVC labels are preserved. Their ServiceMonitor also attaches the node name
and `cluster="homelab"` labels.

## Dashboard Mapping

| Dashboard | Primary metric producers |
| --- | --- |
| Kubernetes / Views / Global | All three |
| Kubernetes / Views / Namespaces | kube-state-metrics and kubelet/cAdvisor |
| Kubernetes / Views / Nodes | node-exporter, kubelet/cAdvisor, and kube-state-metrics |
| Kubernetes / Views / Pods | kube-state-metrics and kubelet/cAdvisor |

The sources complement one another:

- kube-state-metrics says what Kubernetes requested and whether objects are healthy.
- node-exporter says whether the underlying Orange Pi nodes are healthy.
- kubelet/cAdvisor says what pods and containers are actually consuming.
